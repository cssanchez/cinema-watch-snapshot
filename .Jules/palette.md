# Palette's Journal
## 2024-04-07 - Accessibility Polish\n**Learning:** In a static site generated via Python templates, focus styles for keyboard navigation need to use `:focus-visible` rather than `:focus` to maintain good mouse UX, and language switcher buttons without text labels need `aria-label` attributes.\n**Action:** Replaced `:focus` with `:focus-visible` on interactive elements and added `aria-label` to language toggle buttons across all static HTML files.

- **Date:** 2026-04-08
- **Learning:** Context-setting elements like location selectors perform best when extracted from nested dropdowns into prominent, standalone UI elements (e.g., pill buttons) that clearly dictate the state of the content below them.
- **Action:** Created `move_location_selector.py` script to relocate the zone dropdown from the top navigation bar to a dedicated horizontal block underneath the hero banner, using the pre-existing `.location-selector` CSS class.
- **Date:** 2026-04-13\n- **Learning:** Form actions such as "Filter" buttons correctly disable via JavaScript during processing, but lacking specific CSS rules for the `:disabled` pseudo-class leaves them visually indistinguishable from active buttons, causing user confusion.\n- **Action:** Added a transformation step in `apply_ux_improvements.py` to inject CSS for `button:disabled, input:disabled, select:disabled` with `opacity: 0.5; cursor: not-allowed;` to explicitly show the disabled state.
- **Date:** 2026-04-14\n- **Learning:** `<details>`/`<summary>` elements used for expanding/collapsing sections in this codebase require explicit focus states (`:focus-visible`) and expand/collapse visual cues (like a chevron `::after` with rotation based on the `[open]` attribute) to maintain proper accessibility and user experience.\n- **Action:** Added `transform_8_advanced_filters_ux` to `apply_ux_improvements.py` to inject these CSS rules for the "More filters" accordion (`.front-advanced-extra > summary`).
- **Date:** 2026-04-14
- **Learning:** Complex form interactions, like advanced filters, often implement JS logic to handle reset states (e.g., `data-front-advanced-reset="true"`). However, if the corresponding UI element is omitted from the static markup, users cannot discover this functionality, hindering UX and form usability.
- **Action:** Added `transform_9_add_clear_filters_button` to `apply_ux_improvements.py` to append a visible `<button type="button" class="btn btn-secondary" data-front-advanced-reset="true">Clear filters</button>` next to the primary form submission button.
